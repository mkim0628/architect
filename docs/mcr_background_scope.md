# MCR 과제 — 배경 · 필요성 · 범위 (병합안)

## 1. 과제 배경 (이러한 상황이다)

### 1.1 LLM 추론의 메모리 병목이 심화되고 있다

LLM 서비스의 컨텍스트가 길어지면서 decode 단계가 추론 시간의 지배 항목이 되었다. Decode는 매 토큰 생성마다 누적된 KV cache 전체를 읽어야 하는 memory-bandwidth-bound 연산으로, 자체 P/D(prefill/decode) 분리 벤치마크 실측에서 decode 대기 시간이 end-to-end latency의 70–85%를 차지했다. 동시에 KV cache 크기는 컨텍스트 길이와 동시 세션 수에 비례해 증가하며 HBM 용량을 압박한다 — KV가 HBM에 수용되지 못하는 순간 재계산·스왑으로 goodput이 급락한다. 나아가 agent 워크로드는 세션을 넘는 **장기 기억**을 요구하기 시작했다. KV가 요청 수명의 일회성 버퍼가 아니라 세션·사용자 단위로 영속하는 자산이 되면서, 이 병목은 일시적 현상이 아니라 구조적으로 심화되는 추세다.

### 1.2 자사는 memory-centric 디바이스 포트폴리오를 확보했다

자사는 PIM, PNM, Custom HBM, HBF, CXL 기반 메모리 확장 등 memory-centric 제품군을 개발하고 있다. 각 디바이스는 근접 연산, 대역폭, 용량이라는 서로 다른 축의 무기를 제공하며, 조합 시 기존 GPU HBM 단일 체계로는 불가능했던 설계 공간을 연다.

### 1.3 이로 인해 문제 공간 자체가 재정의되었다

디바이스 포트폴리오의 등장은 단순한 선택지 추가가 아니라 시스템 설계의 전제를 두 방향에서 바꾼다.

- **연산기 중심 → 메모리 중심**: PIM/PNM은 "데이터를 연산기로 가져온다"는 폰노이만 전제를 "연산을 데이터가 있는 곳으로 보낸다"로 뒤집는다. 연산 배치·데이터 이동·스케줄링의 기존 가정이 모두 재검토 대상이 된다.
- **메모리 계층의 심화**: HBM–DRAM–SSD 3단 계층이 HBM–HBF–DRAM–CXL Memory–SSD의 5단 이상으로 깊어진다. tier 수가 늘수록 배치 결정의 조합 공간은 기하급수로 커지고, 수동적·정적 tiering의 한계가 명확해진다.

기존 오픈소스 추론 스택(vLLM 등)은 "KV는 GPU HBM에 있다"는 재정의 **이전의** 가정 위에 설계되어 있어, 이 새로운 문제 공간을 담을 수 없다.

## 2. 과제 필요성 (따라서 ~가 필요하다)

### 2.1 계층화된 이종 메모리를 1급 자원으로 다루는 런타임 아키텍처가 필요하다 (← 배경 1.3)

문제 공간이 재정의되었으므로 기존 스택의 점진 개선이 아니라, 깊어진 메모리 계층과 근접 연산을 1급 개념으로 수용하는 런타임 계층의 아키텍처 설계가 필요하다. 디바이스를 파라미터로 취급하고(topology model), 정책과 메커니즘을 분리하며, 실행 프레임워크와의 결합 방식을 명시적 설계 결정(DP)으로 다루어야 — PIM/PNM/HBF 등 로드맵 디바이스가 나올 때마다 재설계하지 않는 구조가 성립한다.

### 2.2 배치×압축×재사용의 joint orchestration이 필요하다 (← 배경 1.1)

Decode-bound 환경에서 KV의 위치와 압축 상태가 곧 서비스 성능이므로, 콜드 데이터를 밀어내는 수동적 tiering만으로는 병목이 해소되지 않는다. KV의 **배치(placement)**, **압축(compression — 품질 비용 수반)**, **재사용(reuse — 비연속 chunk의 selective recompute 판단 수반)** 은 상호 의존적 결정이며, 요청별 SLO와 품질 예산의 관점에서 함께 조율되어야 한다. 장기 기억을 요구하는 agent 워크로드에서는 이 조율이 세션을 넘는 시간 축까지 확장된다.

### 2.3 이 런타임은 자사 디바이스의 레퍼런스 SW 스택이자 E2E 성능 증명의 수단이 된다 (← 배경 1.2)

디바이스는 시스템 관점의 E2E 증명 없이는 가치를 인정받지 못한다. 응용(LLM 서빙)과 디바이스를 연결하는 스택이 곧 런타임이므로, 2.1·2.2를 충족하는 런타임은 자연스럽게 (1) 자사 메모리 제품군의 **레퍼런스 소프트웨어 스택**, (2) GPU HBM 단일 tier 대비 성능 개선을 정량 입증하는 **E2E 증명 vehicle**이 된다. 즉 본 과제의 산출물은 설계 과제의 결과물인 동시에 자사 메모리 사업의 소프트웨어 자산이다.

## 3. 과제 범위 · 목적 (따라서 ~를 할 것이다)

### 3.1 목적

계층화된 이종 메모리 시스템 위에서 LLM 추론의 **goodput@SLO를 극대화**하는 런타임, **MCR(Memory-Centric Runtime)의 아키텍처를 설계**하고, 이를 자사 memory-centric 디바이스의 레퍼런스 스택으로 성립시킨다.

### 3.2 범위 (In Scope)

1. **MCR 레퍼런스 아키텍처 설계** (← 필요성 2.1, 2.3): Inference Orchestration(control plane) / Inference Engine / Memory Engine(data plane)의 3-패키지 구조와 공개 인터페이스(KV Locator, CompressionOp) 정의. 디바이스는 Tier Topology Model의 파라미터로 plug-in되는 구조로 설계하여 로드맵 디바이스 수용성을 확보
2. **핵심 설계 결정(DP) 도출·평가** (← 필요성 2.1, 2.2): DP1 Framework 실행 구조(vLLM 확장형 vs 독립), DP2 KV 배치·압축 관리 주체(중앙 정책 vs 자율) — 후보구조, QA 평가, ADR
3. **E2E 실행 경로 확보 및 정량 성능 증명** (← 필요성 2.2, 2.3): 대표 워크로드(long-context RAG, multiturn, agent memory) 기준 P/D disaggregated 벤치마크가 자사 메모리 tier 위에서 완주하는 E2E 경로를 구축하고, GPU HBM 단일 tier baseline 대비 goodput@SLO·유효 KV capacity·품질 bound 개선을 실측·비용 모델로 입증

### 3.3 범위 외 (Out of Scope)

- **메모리 디바이스 HW 설계 자체** — Tier Topology Model의 파라미터(대역폭·용량·지연)로만 취급
- **모델 학습·압축 알고리즘 자체의 개발** — 압축 mechanism은 기존 기법을 채용하고, 본 과제는 이를 조율하는 구조와 정책의 위치를 다룸
- **클러스터 인프라 provisioning** — desired state 산출까지가 런타임 책임이며, 집행은 인프라 계층에 위임

### 3.4 QA 연결

목적(goodput@SLO)은 QA1로 직결되며, 필요성 2.2는 QA2(메모리 효율)·QA3(응답 품질)로, 필요성 2.1은 QA4(확장성·진화성)로, 필요성 2.3의 레퍼런스 성립 요건은 QA5(개발·운영 비용)와 DP1의 판단 기준으로 계승된다. 상세 정의는 [02_design_points_dp1_dp2.md](02_design_points_dp1_dp2.md) §0 참조.
