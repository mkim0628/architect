# MCR 과제 — 배경 · 필요성 · 범위 (v4)

변경 이력: v4 — 배경 1을 "① AI Agent의 특징(장기 기억 요구→컨텍스트 폭증) →
② 메모리 병목 심화(KV↑, HBM 초과 시 단순 offloading의 성능 저하) → ③ 업계
현황(메모리 제품군 등장 + 자사 포트폴리오)"으로 재구성. 병목의 출발점을
워크로드 변화(Agent화)에서부터 세워 "왜 컨텍스트가 폭증하는가"의 원인 서사를
보강. v3 — 배경 2에 "④ 성능은 런타임이 실현한다" 단을 신설해 ①~⑥ 6단
논리로 확장. 기존 ④(현 런타임 한계) 앞에 런타임 계층 자체의 중요성(결정
계층으로서의 역할, 결정 공간의 폭발, 동일 HW 실측 격차, 실현 계층으로서의 사업
함의)을 먼저 세워 "왜 갑자기 런타임인가"의 논리 비약을 해소. vLLM·FlexGen
실측은 ④의 근거로 이동. v2 — 배경을 ①~⑤ 5단 논리(배경1: 병목→업계 해법→자사
재료 / 배경2: 현 런타임 한계→필요성)로 재구성. "승부처의 이동" 추상 명제를
"현재 런타임의 한계"라는 검증 가능한 사실 명제로 대체하고, vLLM·FlexGen 실측
근거와 LMCache 비교를 편입. v1 — 배경·필요성·범위 병합안 최초 작성.

## 1. 과제 배경 1 — Agent 워크로드, 병목 심화, 그리고 업계의 재료

### 1.1 ① AI Agent의 특징 — 워크로드가 장기 기억을 요구한다

AI 활용의 중심이 단발 Q&A 챗봇에서 Agent로 이동하고 있다. Agent 워크로드에는
다음의 구조 변화가 필수적으로 일어난다:

- **Multi-turn loop이 기본 패턴**: 하나의 요청이 한 번의 추론으로 끝나지 않고,
  계획 → 실행 → 관찰 → 재계획의 루프를 수십~수백 스텝 반복한다.
- **툴 호출 기본 장착**: 매 스텝이 검색·코드 실행·외부 API 등 툴을 호출하고,
  그 결과물(문서·로그·데이터)이 컨텍스트로 계속 유입된다.
- **Sub-agent 시스템**: 하나의 목적을 위해 여러 sub-agent가 분업·병렬 실행하고
  산출물을 모으는 구조가 표준이 되고 있다.

세 변화의 공통 귀결은 **장기 기억 요구**다. 루프의 매 스텝이 이전 대화·툴
결과·sub-agent 산출물을 참조해야 하고, 이 기억은 세션을 넘어 사용자·과제
단위로 쌓인다. 즉 Agent화는 **컨텍스트 길이의 폭증**이라는 문제를 반드시 풀
것을 요구한다.

### 1.2 ② 이 변화가 메모리 병목을 심화시킨다

①의 컨텍스트 폭증은 추론의 지배 항목을 연산에서 메모리로 옮긴다.

- **컨텍스트 증가 → KV cache 증가**: KV cache는 컨텍스트 길이 × 동시 세션 수에
  비례해 커진다. 게다가 decode는 매 토큰마다 누적된 KV 전체를 읽는
  memory-bandwidth-bound 연산이라(자체 P/D 분리 벤치 실측에서 decode 대기가
  end-to-end latency의 70–85%(A)) KV 증가는 용량과 대역폭을 동시에 압박한다.
- **HBM 초과 + 단순 offloading의 성능 저하**: 커진 KV는 HBM 용량을 초과하기
  시작한다. 이를 단순히 DRAM→SSD로 내리는 offloading은 tier마다 대역폭이
  ~10×씩 낮아지는 계단(HBM ~TB/s → DRAM ~100GB/s → SSD ~10GB/s)을 그대로
  노출해, 수동적 스왑만으로는 latency 급증·처리량 급락으로 이어진다.

### 1.3 ③ 업계 현황 — 메모리 중심 제품군의 등장

- **메모리 업계의 대응**: 이 병목을 하드웨어 차원에서 풀기 위한 제품군이
  등장하고 있다. 대역폭 축에는 HBM 고도화와 **근접 연산(PIM/PNM)** — 데이터를
  연산기로 가져오는 대신 연산을 데이터가 있는 곳으로 보내 이동량 자체를 줄인다.
  용량 축에는 **CXL 기반 확장·풀링과 새로운 중간 tier(HBF 등)** — HBM 밖의 크고
  싼 계층으로 용량을 확장한다. 메모리 계층은 5+ tier로 깊어지는 중이다.
- **자사(삼성)의 제품군**: 근접 연산(PIM·CIM 계열)부터 계층화(Custom
  HBM·HBF·CXL 기반 확장)까지, 두 축 모두에 걸친 memory-centric 제품
  포트폴리오를 보유·개발 중이다. 각 디바이스는 근접연산·대역폭·용량이라는 서로
  다른 축의 무기를 제공하며, 조합하면 GPU HBM 단일 체계로는 불가능했던 설계
  공간이 열린다. **즉 병목을 풀 하드웨어 재료는 갖춰지고 있다.**

## 2. 과제 배경 2 — 성능을 실현할 런타임이 없다

### 2.1 ④ AI 성능은 하드웨어가 아니라 런타임이 실현한다

②의 병목과 ③의 재료 사이에는 결정의 공백이 있다. 하드웨어는 성능의
**능력(capability)** 을 제공할 뿐, 성능 자체를 만들지 못한다.

- **런타임 = 성능을 결정하는 계층**: 모델(응용)은 "무엇을 계산할지"를, 디바이스는
  "무엇이 가능한지"를 정의할 뿐이다. 그 사이 — 어떤 데이터를 어디에 두고(배치),
  언제 옮기고(이동), 어떤 연산을 어디서 실행하고(연산 배치), 어떤 요청부터
  처리할지(스케줄링) — 는 하드웨어가 스스로 답할 수 없는 질문이며, 이 결정을
  내리는 계층이 런타임이다. 시스템이 실제로 도달하는 성능(goodput·latency·비용)은
  이 결정의 질로 정해진다.
- **결정 공간의 폭발 — 런타임의 비중이 커지는 이유**: GPU+HBM 단일 체계에서는 둘
  곳도 연산할 곳도 하나뿐이라 결정이 자명했고, 런타임은 얇아도 됐다. 그러나 ②로
  병목이 메모리로 이동하고 ③의 계층화·근접 연산 디바이스가 들어오는 순간
  선택지는 tier × 디바이스 × 요청의 조합 공간으로 폭발하며, 데이터 배치와 연산
  배치라는 두 결정은 서로 결합(coupled)되어 있고, 잘못된 정책은 baseline보다도
  느리다. **하드웨어가 이종화될수록 성능의 분산은 런타임 쪽으로 이동한다.**
- **실측 근거 — 같은 하드웨어, 런타임만으로 갈린 성능**: 메모리 관리
  방식(paging)을 고친 것만으로 같은 GPU에서 KV cache 낭비가 60–80% → <4%로
  줄고 처리량 2–4×가 갈렸으며(vLLM, SOSP'23), 같은 16GB GPU 한 장에서
  offloading 배치 정책만으로 최대 처리량 **100×** 차이가 났다(FlexGen, ICML'23
  vs Accelerate·ZeRO-Inference). **하드웨어 스펙은 성능의 상한을 정할 뿐, 실제
  도달 성능은 런타임이 정한다**는 실증이다.
- **사업 함의 — 디바이스 가치의 유일한 통로**: 디바이스의 가치는 런타임을
  통해서만 응용에 도달한다. 런타임이 지원하지 않는 하드웨어는 응용에서 보이지
  않는다 — GPU 생태계의 해자가 실리콘이 아니라 CUDA 계열 SW 스택이라는 것이
  방증이다. 따라서 ③의 재료가 성능이 되려면 런타임이라는 관문을 통과해야 한다.

### 2.2 ⑤ 그런데 현재의 런타임은 연산기 중심이다 — 이 재료를 활용하지 못한다

④에서 성능의 관문이 런타임임을 확인했다. 그런데 현재 런타임 지형을 보면 그
관문이 모두 연산기(GPU)를 중심으로 설계되어 있다:

- **GPU 중심 설계**: vLLM 등 기존 스택은 "연산은 GPU, KV는 GPU HBM"이라는 단일
  메모리 가정 위에 설계되어 있다. 메모리는 GPU의 부속물로 취급되어 PIM/PNM 같은
  근접 연산 디바이스는 표현할 자리가 없고, HBM 용량을 넘는 순간
  preemption·재계산·스왑으로 성능이 급락하며, offloading이 있어도 콜드 데이터를
  밀어내는 정적·수동적 부가 기능 수준이다. ④의 실측(KV 60–80% 낭비)이 보여주듯
  메모리를 1급으로 다루지 않는 런타임의 낭비는 이미 단일 tier에서도 크다 —
  이종 tier에서는 그 격차가 100× 급으로 벌어진다.
- **KV 계층의 등장과 그 한계**: 업계도 이 한계를 인지해 KV를 HBM 밖으로 내리는
  계층이 등장했다 — LMCache는 KV를 CPU DRAM·SSD·원격 저장소(Redis 등) 계층에
  영속화·재사용하고 압축(CacheGen)·비접두 재사용(CacheBlend)까지 갖춘, vLLM 공식
  KV offloading connector의 대표 구현이다. 그러나 이들 계층조차 **commodity
  저장소를 수동적 put/get 백엔드로 추상화**할 뿐이다: 디바이스의
  대역폭·지연·용량 특성을 인지하는 배치(topology-aware placement)도, 연산을
  데이터로 보내는 근접 연산도, 요청별 SLO·품질 예산 기반의 정책도 없다 — 데이터
  이동만 있고 연산 배치가 없으므로 대역폭 병목 축은 다루지 못한다.

즉 문제 인식(KV는 HBM 밖으로)은 업계 공통이 되었으나, **자사 메모리 디바이스가
1급 자원으로 들어갈 자리는 어느 런타임에도 없다.**

### 2.3 ⑥ 따라서 자사 메모리 제품군을 활용하는 런타임 개발이 필요하다

③의 재료, ④의 관문, ⑤의 공백을 합치면 결론은 하나다: 자사 메모리 포트폴리오를 GPU의
부속이 아닌 **1급 자원으로 다루며, 데이터 배치·이동·압축·재사용과 연산 배치를
요청별 SLO·품질 예산 기준으로 조율하는 메모리 중심 런타임, MCR(Memory-Centric
Runtime)** 을 직접 개발해야 한다. 이 런타임은 동시에:

1. 자사 메모리 제품군의 **레퍼런스 소프트웨어 스택** — 응용(LLM 서빙)과
   디바이스를 잇는 실현 계층
2. GPU HBM 단일 tier 대비 성능 개선을 정량 입증하는 **E2E 증명 수단**

이 된다. 즉 본 과제의 산출물은 설계 과제의 결과물인 동시에 자사 메모리 사업의
소프트웨어 자산이다.

## 3. 과제 범위 · 목적 (따라서 ~를 할 것이다)

### 3.1 목적

계층화된 이종 메모리 시스템 위에서 LLM 추론의 **goodput@SLO를 극대화**하는
런타임, **MCR(Memory-Centric Runtime)의 아키텍처를 설계**하고, 이를 자사
memory-centric 디바이스의 레퍼런스 스택으로 성립시킨다.

### 3.2 범위 (In Scope)

1. **MCR 레퍼런스 아키텍처 설계** (← 배경 ⑤·⑥): Inference Orchestration(control
   plane) / Inference Engine / Memory Engine(data plane)의 3-패키지 구조와 공개
   인터페이스(KV Locator, CompressionOp) 정의. 디바이스는 Tier Topology Model의
   파라미터로 plug-in되는 구조로 설계하여 로드맵 디바이스 수용성을 확보
2. **핵심 설계 결정(DP) 도출·평가** (← 배경 ⑤·⑥): DP1 실행 스택 소싱 —
   Inference·Memory Engine의 외부 채택 vs 자체 구현(외부 스택 활용형[변형:
   vLLM 확장 / LMCache 편승] vs 자체 구현형), DP2 KV 배치·압축 관리 주체(중앙
   정책 vs 자율) — 후보구조, QA 평가, ADR
3. **E2E 실행 경로 확보 및 정량 성능 증명** (← 배경 ①·②·⑥): 대표 워크로드
   (long-context RAG, multiturn, agent memory) 기준 P/D disaggregated 벤치마크가
   자사 메모리 tier 위에서 완주하는 E2E 경로를 구축하고, GPU HBM 단일 tier
   baseline 대비 goodput@SLO·유효 KV capacity·품질 bound 개선을 실측·비용 모델로
   입증

### 3.3 범위 외 (Out of Scope)

- **메모리 디바이스 HW 설계 자체** — Tier Topology Model의 파라미터(대역폭·용량·지연)로만 취급
- **모델 학습·압축 알고리즘 자체의 개발** — 압축 mechanism은 기존 기법을 채용하고, 본 과제는 이를 조율하는 구조와 정책의 위치를 다룸
- **클러스터 인프라 provisioning** — desired state 산출까지가 런타임 책임이며, 집행은 인프라 계층에 위임

### 3.4 QA 연결

목적(goodput@SLO)은 QA1로 직결되며, ⑥의 조율 요구(배치×압축×재사용)는
QA2(응답 품질)·QA3(메모리 효율)로, ⑥의 1급 자원 아키텍처 요구는
QA4(확장성·진화성)로, 레퍼런스 스택 성립 요건은 QA5(개발·운영 비용)와 DP1의
판단 기준으로 계승된다. 상세 정의는 [00_qa_definitions.md](00_qa_definitions.md)
참조.
